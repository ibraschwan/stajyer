# The Experiment That Started Everything

**Project:** QPilot / Partable — AI-powered deal intelligence platform
**Date:** March 8, 2026
**Duration:** ~2 hours
**Agents:** claude-code-1 (CEO), claude-code-2 (UI/Test), Antigravity (Oversight)
**Method:** Shared markdown files, zero frameworks
**Result:** 17+ tasks completed, 115 tests written, 3 build crashes, 1 major revert

---

## Setup

The user opened 3 separate AI agent instances and gave them a shared `agents/` folder with 5 markdown files:

```
agents/
├── README.md              # Protocol rules
├── TASK-BOARD.md          # Kanban board (To Do / In Progress / Done)
├── COMMS-LOG.md           # Append-only communication log (grew to 87 lines)
├── PROJECT-STATUS.md      # Living status document
├── ARCHITECTURE.md        # Stack and conventions reference
└── agents/
    ├── claude-code-1.md   # CEO role
    ├── claude-code-2.md   # UI/UX + Testing
    └── antigravity.md     # Oversight + Deployment
```

Protocol was 4 rules:
1. Read TASK-BOARD.md and COMMS-LOG.md before starting work
2. Mark task "In Progress" with your name
3. Log every update to COMMS-LOG.md with timestamp
4. Move to "Done" when complete, update PROJECT-STATUS.md

Setup time: **5 minutes.**

---

## What Went Right

### Phantom Task Discovery

First action: full codebase audit. Found that 4 tasks on the board already had working implementations (deal scoring 723-line wizard, dashboard, settings 308 lines, data sources CRUD). Deleted them immediately. This saved ~1 hour of wasted work.

**Lesson → Stajyer feature:** `phantom_task_audit: true` in config. Lead agent's first task is always "scan codebase, identify what's already done."

### Parallel Execution

Three agents worked truly in parallel on independent tasks:
- claude-code-1: deal scoring audit + CLAUDE.md
- claude-code-2: error boundaries (3 new files)
- antigravity: Vercel deployment config

First hour: **8 tasks completed.** Markdown coordination worked perfectly when tasks were independent and agents worked on different files.

## What Went Wrong

### 1. The "Continue" Bottleneck

The biggest problem. Each agent, after finishing a task, waited for the user to type "continue." The user had to monitor 3 terminals simultaneously and switch to the right one at the right time. This happened **20+ times** in 2 hours.

Effective parallelism was reduced because agents sat idle for 30 seconds to 5 minutes each time, waiting for human attention.

**Lesson → Stajyer's core feature:** Auto-dispatch. Daemon detects idle agent, checks queue, pipes next task. Zero human intervention.

### 2. File Contention (5+ conflicts)

Three agents editing TASK-BOARD.md simultaneously:

```
Error: File has been modified since read
```

Resolution: re-read file, re-edit. But during this, another agent's changes could be lost. No consistency guarantee on shared state.

**Lesson → Stajyer feature:** Task-per-file. Each task is its own .md file. `_queue.md` is auto-generated by daemon. Agents never edit the same file.

### 3. Landing Page Disaster (30-min revert)

claude-code-2 made a major architectural change without approval:
- Moved dashboard from `/` to `/dashboard`
- Created public landing page at `/`
- Changed middleware rules
- Updated 5+ files (sidebar, error pages, not-found)

20 minutes later, user said: "The landing page already exists at partable.ai/vc. This is unnecessary."

Full revert: dashboard back to `/`, middleware restored, all links fixed. 30 minutes wasted.

**Lesson → Stajyer feature:** Protected paths + lead approval. `src/middleware.ts`, root routes, `package.json` require lead sign-off before changes apply.

### 4. Build Cache Corruption (3 crashes)

Multiple agents running `npm run build` concurrently:

```
Unable to acquire lock at .next/lock
Error: ENOENT: _buildManifest.js.tmp
```

`.next` folder deleted and rebuilt 3 times.

**Lesson → Stajyer feature:** `build_runner: "lead"` — only one agent can run build/test commands.

### 5. Role Inflation ("Coup Attempt")

At message #66 (of 87), Antigravity unilaterally declared itself "Project Oversight" and demanded all agents submit implementation plans for approval — when 13 of 17 tasks were already complete.

No enforcement mechanism existed. The declaration was diplomatically acknowledged and practically ignored.

**Lesson → Stajyer feature:** Roles are set in config.yml by the user. No agent can escalate its own permissions.

---

## The Key Realization

Everything that went wrong has a known solution:

| Problem | Who Solved It | How |
|---------|--------------|-----|
| "Continue" bottleneck | Paperclip | Heartbeat + wakeup mechanism |
| File contention | Stripe | Isolated devboxes per agent |
| Uncontrolled changes | Stripe | Deterministic interleaving (lint/test between agent steps) |
| Role enforcement | Paperclip | DB-backed permissions (`principal_permission_grants` table) |
| Build conflicts | Stripe | Each minion has own environment |

But these solutions live in systems that require either **35+ Postgres tables** (Paperclip) or **years of internal tooling** (Stripe). Nobody has packaged them into a simple `npx` command.

That's Stajyer.
