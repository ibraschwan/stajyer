# Technical Architecture

## Overview

Stajyer has 5 layers, each solving a specific problem discovered in the QPilot experiment:

| Layer | Problem It Solves | Inspired By |
|-------|------------------|-------------|
| 1. Project Structure | File contention | CRDT-like task-per-file |
| 2. Daemon | "Continue" bottleneck | Paperclip heartbeat |
| 3. Auto-Dispatch | Agent idle time | Paperclip wakeup mechanism |
| 4. Blast Radius Guard | Uncontrolled changes | Stripe deterministic interleaving |
| 5. Adapter System | Agent lock-in | Paperclip adapter pattern |

---

## Layer 1: Project Structure (Convention over Configuration)

```bash
npx stajyer init
```

Creates:

```
.stajyer/
├── config.yml              # Agent definitions, rules, protected files
├── tasks/
│   ├── 001-error-boundaries.md    # One file per task
│   ├── 002-api-tests.md
│   └── _queue.md                  # Auto-generated queue summary
├── comms/
│   ├── general.md                 # General communication
│   └── _feed.md                   # Auto-generated feed
├── agents/
│   ├── lead.md                    # Lead agent profile + rules
│   ├── dev-1.md                   # Worker 1 profile
│   └── dev-2.md                   # Worker 2 profile
├── rules/
│   ├── global.md                  # Rules for all agents
│   ├── components.md              # Rules for src/components/
│   └── api.md                     # Rules for src/app/api/
└── daemon.pid                     # Daemon process ID
```

### Why task-per-file?

In our experiment, 3 agents editing TASK-BOARD.md caused 5+ conflicts. With task-per-file:
- Each agent only writes to its own task file (update status, add notes)
- `_queue.md` is auto-generated by daemon — never edited by agents
- Zero file contention

### Task file format

```markdown
---
id: "001"
title: "Add error boundaries"
status: "done"           # pending | assigned | in_progress | done | failed | needs_human
assigned_to: "dev-1"
created_by: "lead"
session_id: "abc123"     # Claude Code session ID for --resume
created_at: "2026-03-08T20:15:00Z"
completed_at: "2026-03-08T20:35:00Z"
tokens_used: 12400
---

# Add error boundaries

Create error.tsx, global-error.tsx, and not-found.tsx for React error handling.

## Requirements
- global-error.tsx: catches root layout errors, full HTML shell with retry
- error.tsx: catches page-level errors, retry + dashboard link
- not-found.tsx: custom 404

## Agent Notes
- Created 3 files, all type-check clean
- Matches project styling (#1a1a2e buttons, gray-50 bg)
```

Frontmatter = structured data (daemon parses with gray-matter).
Body = human-readable context (agent reads as instructions).

## Layer 2: Daemon

```bash
npx stajyer up
```

Single Node.js process. Components:

```
stajyer daemon
│
├── File Watcher (chokidar)
│   ├── .stajyer/tasks/*.md     → Task created? Status changed? Completed?
│   ├── .stajyer/comms/*.md     → New message?
│   └── src/**/*                → Protected file changed?
│
├── Process Manager
│   ├── lead: claude --print --output-format stream-json --session-id ...
│   ├── dev-1: claude --print --output-format stream-json --session-id ...
│   └── dev-2: codex --run "..."
│
├── Event Bus (EventEmitter, in-memory)
│   ├── task:created      → Route to available agent
│   ├── task:completed    → Update _queue.md, notify lead
│   ├── agent:idle        → Check queue, auto-assign
│   ├── agent:crashed     → Restart with --resume using saved session_id
│   └── file:protected    → Stash change, ask lead for approval
│
└── State (in-memory, synced to markdown)
    ├── Agent states: idle | working | waiting | crashed
    ├── Task assignments: which agent owns which task
    ├── Session IDs: for --resume on crash recovery
    └── Token counters: per-agent usage
```

### Why no database?

Paperclip uses 35+ Postgres tables because it supports:
- Multi-company (we don't need — single project)
- Multi-user (we don't need — single developer)
- Persistent cross-session state (markdown files persist)
- Complex queries (we just scan a directory)

Our state source of truth is always the markdown files. Daemon's in-memory state is a cache that rebuilds on restart by scanning `.stajyer/tasks/` and `.stajyer/agents/`.

### Process management

Each agent is a child process spawned with `node:child_process.spawn()`:

```typescript
// Simplified
const agent = spawn('claude', [
  '--print',
  '--output-format', 'stream-json',
  '--resume', sessionId,        // Resume previous session if exists
  '--max-turns', '50',
], {
  cwd: projectRoot,
  env: { ...process.env, ...agentEnv },
  stdio: ['pipe', 'pipe', 'pipe'],  // stdin, stdout, stderr all piped
});
```

Stdin: daemon writes prompts (task instructions + context).
Stdout: daemon reads stream-json events (progress, result, usage).
Stderr: daemon reads error messages.

## Layer 3: Auto-Dispatch

The core innovation. When an agent finishes a task:

```
Agent stdout emits 'result' event
  → Daemon marks task as "done" in task file frontmatter
  → Daemon checks _queue.md for next task assigned to this agent
    → Found: pipe new task prompt to agent's stdin
    → Not found: check for unassigned pending tasks
      → Found: assign to agent, pipe prompt
      → Not found: notify lead agent "dev-1 is idle, assign new task or say 'idle'"
        → Lead creates new task file → daemon detects → assigns → pipes to dev-1
```

Time from agent-idle to next-task: **< 3 seconds** (vs 30s-5min with manual "continue").

### Lead agent as auto-coordinator

When all workers are idle and no tasks remain, the lead agent is prompted:

```
"All workers are idle. No pending tasks.
Completed tasks: [list of done tasks with summaries]
What's next? Create new task files or say 'all done'."
```

This creates the autonomous CEO cycle from our experiment — but without the user being the message broker.

## Layer 4: Blast Radius Guard

### File ownership

```yaml
# .stajyer/config.yml
agents:
  lead:
    adapter: claude-code
    model: claude-opus-4-6
    role: "Coordination, code review, architecture"
    owns: []  # Lead doesn't write code, coordinates

  dev-1:
    adapter: claude-code
    model: claude-sonnet-4-6
    role: "Frontend: components, pages, styling, frontend tests"
    owns:
      - "src/components/**"
      - "src/app/**/page.tsx"
      - "src/__tests__/**/*.test.tsx"

  dev-2:
    adapter: claude-code
    model: claude-sonnet-4-6
    role: "Backend: API routes, middleware, database, backend tests"
    owns:
      - "src/app/api/**"
      - "src/lib/**"
      - "src/__tests__/**/*.test.ts"
```

### Protected paths

```yaml
protected:
  - path: "src/middleware.ts"
    requires: "lead"              # Lead must approve
  - path: "package.json"
    requires: "build-pass"        # Build must pass after change
  - path: ".env*"
    requires: "never"             # No agent can touch

build_runner: "lead"              # Only lead runs build/test
```

### How protection works

Daemon watches all project files via chokidar. When an agent modifies a protected file:

1. Daemon detects the change (fs.watch event)
2. Stashes the change (git stash or temp copy)
3. Sends lead agent a message: "dev-2 wants to modify src/middleware.ts. Diff: [change]. Approve? (yes/no/modify)"
4. Lead approves → change applied
5. Lead rejects → change discarded, dev-2 notified with reason

### Build checkpoints (inspired by Stripe)

```yaml
rules:
  max_ci_rounds: 2              # At most 2 build/test cycles per task
  local_lint_on_save: true      # Lint after every file save (<5s)
  require_build_on_complete: true  # Build must pass before task is "done"
```

Stripe: "Shift feedback left. Best for humans and agents if any lint step that would fail in CI is enforced on git push."

## Layer 5: Adapter System

### Interface

```typescript
interface StajyerAdapter {
  type: string;                           // "claude-code" | "codex" | "custom"

  spawn(config: AgentConfig): AgentProcess;
  sendTask(process: AgentProcess, task: TaskFile): void;
  parseOutput(chunk: Buffer): AgentEvent[];
  isIdle(process: AgentProcess): boolean;
  resume(process: AgentProcess, sessionId: string): void;
  getUsage(process: AgentProcess): TokenUsage;
  terminate(process: AgentProcess): Promise<void>;
}

interface AgentEvent {
  type: "progress" | "result" | "error" | "tool_use" | "idle";
  content?: string;
  usage?: { inputTokens: number; outputTokens: number };
  sessionId?: string;
}
```

### Built-in adapters

**claude-code:**
- Spawns: `claude --print --output-format stream-json`
- Parses stream-json for events, usage, session ID
- Supports `--resume`, `--model`, `--max-turns`
- Detects "waiting for input" state → signals idle

**codex:**
- Spawns: `codex --run`
- Parses output for completion signals

**custom:**
- Spawns any CLI command defined in config
- Expects JSON lines on stdout: `{"type": "result", "content": "..."}`
- Stdin receives task prompt as plain text

### Adding a new adapter

```typescript
// adapters/my-agent.ts
import { StajyerAdapter } from '@stajyer/core';

export const myAgent: StajyerAdapter = {
  type: 'my-agent',
  spawn(config) { /* ... */ },
  sendTask(proc, task) { /* ... */ },
  parseOutput(chunk) { /* ... */ },
  // ...
};
```

Register in config:

```yaml
agents:
  researcher:
    adapter: my-agent
    config:
      command: "python my_agent.py"
```

## Tech Stack

```
Runtime:       Node.js 20+
Language:      TypeScript (strict)
Build:         tsup (ESM + CJS dual output)
CLI framework: commander.js
Terminal UI:   ink (React for CLI)
File watching: chokidar
Process mgmt:  node:child_process
Markdown:      gray-matter (frontmatter) + remark (parsing)
Testing:       vitest
Package:       npm / npx
```

Target: < 15 runtime dependencies. Cold start < 2 seconds.
